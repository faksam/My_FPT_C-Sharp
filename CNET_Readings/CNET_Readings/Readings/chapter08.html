In most programs, it is quite common for objects in a system to engage in a two-way conversion through the use of callback interfaces, events and other programming constructs. To set the stage, this chapter begins by examining how interface types may be used to enable callback functionality.
Next, you learn about the .NET delegate type, which is a type-safe object that "point to" other method(s) that can be invoked at a later time. .NET delegates are objects that have built-in support for multicasting and asynchronous method invocation.
One you learn how to create and manipulate delegate types, you then investigate the C# event keyword, which simplifies and streamlines the process of working with delegate types. Finally, this chapter examines new delegate-and-event-centric language features of C#, including anonymous methods and method group conversions.

Understanding the .NET Delegate type

A delegate is a type-safe object that points to another method (or possibly multiple methods) in the application, which can be invoked at a later time. Special speaking, a delegate type maintains three important pieces of information:

  • The name of the method on which it makes calls
  • The arguments (if any) of this method
  • The return value (if any) of this method

Once a delegate has been created and provided the aforementioned information, it may dynamically invoke the method(s) it points to at runtime. As you will see, every delegate in the .NET Framework is automatically endowed with the ability to call their methods synchronously or asynchronously.

Defining a Delegate in C#

When you want to create a delegate in C#, you make use of the delegate keyword. The name of your delegate can be whatever you desire. However, you must define the delegate to match the signature of the method it will point to. For example, assume you wish to build a delegate name BinaryOp that can point to any method that returns an integer and takes two integers as input parameters:
public delegate int BinaryOp(int x, int y);
When the C# compiler processes delegate types, it automatically generates a sealed class deriving from System.MulticastDelegate. This class provides the necessary infrastructure for the delegate to hold onto the list of methods to be invoked at a later time.
Delegates can also "point to" methods that contain any number of out or ref parameters. For example, assume you have defined a delegate type that can point to any method returning a string and receiving three System.Boolean input parameters:
public delegate string MyOtherDelegate(out bool a, ref bool b, bool c);

A simple Delegate example

namespace SimpleDelegate
{
     //This delegate can point to any method, taking two integers and returning an integer.
     public delegate int BinaryOp(int x, int y);

     //This class containts methods BinaryOp will point to.
     public class SimpleMath
     {
          public static int Add(int x, int y)
          { return x + y; }
          public static int Subtract(int x, int y)
          { return x - y; }
    }

    class Program
    {
        Console.Writeline("***** Simple Delegate Example *****");

        //Create a BinaryOp object that "point to" SimpleMath.Add().
        BinaryOp b = new BinaryOp(SimpleMath.Add());
        //Invoke Add() method using delegate.
        Console.WriteLine("10 + 10 is {0}", b(10,10));
        Console.ReadLine();
    }
}

Multicast Delegates

.NET delegates have the intrinsic ability to multicast. In other words, a delegate object can maintain a list of methods to call, rather than a single method. When you wish to add multiple methods to a delegate object, you simply make use of the overloaded += operator, rather than a direct assignment. In terms of CIL code, the += operator resolves to a call to the static Delegate.Combine() method (you could call Delegate.Combine() directly, but the += operator offers a simpler alternative).
The Delegate class also defines a static Remove() method that allows a caller to dynamically remove a member from the invocation list. C# developers can leverage the overload -= as a shorthand notation.

Understanding Delegate Covariance

Covariance allows you to build a single delegate that can point to methods returning class types related by classical inheritance.
Assume you have two classes: Car and SportsCar derived from Car. We can define a single delegate that may return a Car or SportsCar. The below example illustrates this delegate.

class Program
{
     public delegate Car ObtainVehicalDelegate();

     public static Car GetBasicsCar()
     { return new Car(); }

     public static SportsCar GetSportsCar()
     { return new SportsCar(); }

     static void Main()
     {
          Console.WriteLine("***** Delegate Covariance *****");
          ObtainVehicalDelegate targetA = new ObtainVehicalDelegate(GetBasicsCar);
          Car c = targetA();

          //Covariance allows this target assignment.
          ObtainVehicalDelegate targetB = new ObtainVehicalDelegate(GetSportsCar);
          SportsCar sc = (SportsCar) targetB();
          Console.ReadLine();
     }
}

Understanding C# Events

Delegates are fairly interesting constructs in that they enable two objects in memory to engage in a two-way conversation. However, working with delegates in the raw does entail a good amount of boilerplate code (definding the delegate, declaring necessary member variables, and creating custom registration/unregistration methods).
Because the ability for one object to call back to another object is such a helpful construct, C# provides the event keyword to lessen the burden of using delegates in the raw. When the compiler processes the event keyword, you are automatically provided with registration and unregistration methods as well as any necessary member variable for your delegate types.
Defining an event is a two-step process. First, you need to define a delegate that contains the method to be called when the event is fired. Next, you declare the events (using the C# event keyword) in term of the related delegate. In a nutshell, defining a type that can send events entails the following pattern (show in pseudo-code):

public class SenderOfEvents
{
     public delegate retval AssociateDelegate(args);
     public event AsscociateDelegate NameOfEvent;
}

Sending an event to the caller is as simple as specifying the event by name as well as any required parameters as defined by the associated delegate. To ensure that the caller has indeed registered with event, you will want to check the event against a null value before invoking the delegate's method set.

Summary

In this chapter, you have examined a number of ways in which multiple objects can partake in a bidirectional conversion. First, you examined the use of callback interfaces, which provide a way to have object B make calls on object A through a common interface type.
Next, you examined the C# delegate keyword, which is used to indirectly construct a class derived from System.MulticastDelegate. A delegate is simplify an object that maintains a list of methods to call when told to do so.
Finally, you examined the C# event keyword which, when used in conjunction with a delegate type, can simplify the process of sending your event notifications to awaiting callers.
