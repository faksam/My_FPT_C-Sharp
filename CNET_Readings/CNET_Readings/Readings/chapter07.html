The point of this chapter is examining interface-based programming. Here you learn how to use C# to define and implement interfaces, and come to understand the benefits of building types that support "multiple behaviors".
The remainder of this chapter is spent examining a number of interfaces defined within the .NET class libraries. To showcase how interfaces are leveraged in the .NET base class libraries, this chapter will also examine numerous predefined interfaces implemented by various collection classes defined by the System.Collections namespace.

Defining Interfaces in C#

An interface is a named collection of semantically related abstract members. The specific members defined by an interface depend on the exact behavior it is modeling. An interface expresses a behavior that a given class or structure may choose to support.
At a syntactic level, an interface is defined using the C# interface keyword. Unlike other .NET types, interfaces never specify a base class and contain members that do not take an access modifier (as all interface members are implicitly publish).
Here is a custom interface defining in C#:

//This is interface defines the behavior of "having points."
public interface IPointy
{
     //Implicitly public and abstract
     byte GetNumberOfPoints();
}

Implementing an Interface in C#

When a class (or structure) chooses to extend its functionality by supporting interface types, it does so using a comma-delimited list in the type definition. Be aware that the direct base class must be the first item listed after the colon operator.
Understand that implementing an interface is an all-or-nothing proposition. The supporting type is not able to selectively choose which members it will implement. For example, if you are implementing an interface that defines ten members, the type is now responsible for fleshing out the details of the ten abstract entites.
Here is the implementation of the new Triangle class type.

public class Triangle : IPointy
{
     public Triangle() {}
     //IPointy Implementation.
     public byte Points
     {
          get {return 3;}
     }
}

Contrasting Interfaces to Abstract base classes

Like an interface, when a class derives from an abstract base class, it is also under obligation to flesh out details of the abstract methods. However, abstract base classes do far more than define a group of abstract methods. They are free to define public, private and protected state data, as well as any number of concrete methods that can be accessed by the subclasses.
Interfaces, on the other hand, are pure protocol. Interfaces never define state data and never provide an implementation of the methods.
The interface-based protocol allows a given type to support numerous behaviors, while avoiding the issues that arise when deriving from extending multiple base classes.
Most importantly, interface-based programming provides yet another way to inject polymorphic behavior into a system. If multi classes (or structures) implement the same interface in their unique ways, you have the power to treat each type in the same manner.

Summary

An interface can be defined as a named collection of abstract members. Because an interface does not provide any implementation details, it is common to regard an interface as a behavior that may be supported by a given type. When two or more classes implement the same interface, you are able to treat each type the same way.
C# provides the interface keyword to allow you to define a new interface. A type can support as many interfaces as necessary using a comma-delimited list. Furthermore, it is permissible to build interfaces that derive from multiple base interfaces.
Finally, you spent some time investigating the stock collection classes defined within the System.Collections namespace and examining a number of common interfaces used by the collection-centric types.
